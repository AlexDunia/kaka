24. RECURRENCE STRUCTURE
============================================================

Utility file:
src/utils/recurrenceSchedule.js

Weekday values:
- 0: Sunday
- 1: Monday
- 2: Tuesday
- 3: Wednesday
- 4: Thursday
- 5: Friday
- 6: Saturday

Daily recurrence:
- Uses the first occurrence start time and duration.
- Repeats every calendar day.
- A daily event duration of 24 hours or more is treated as overlapping.

Weekly recurrence:
- Uses selected weekday numbers.
- The first occurrence weekday is kept selected.
- All weekly occurrences share the same start time and duration.

Monthly recurrence:
- day_of_month mode repeats on the same calendar day.
- If the target day does not exist in a shorter month, it clamps to the last day of that month.
- nth_weekday mode repeats on patterns like "third Sunday".
- If a fifth weekday does not exist in a month, that month is skipped.

Series ending:
- on_date: stops at the end of the selected date.
- after_occurrences: stops after the selected count.

Preview:
- generateOccurrences() creates preview dates.
- scheduleHasOverlap() detects if the event duration runs into the next occurrence.
- These frontend preview dates are informational. The backend should generate authoritative occurrences server-side.

26. CREATE EVENT API FLOW
============================================================

Frontend API setup:
src/api/axios.js

API base:
- baseURL: /api
- Uses credentials.
- Vite proxies /api to http://127.0.0.1:8000 in development.
- Vite also proxies /sanctum to http://127.0.0.1:8000.

CSRF:

Create-event service calls:
- createEventDraft(form):
  - ensure CSRF cookie.
  - POST /api/create-event with buildCreateEventPayload(form).

- updateEventDraft(eventId, form):
  - ensure CSRF cookie.
  - PATCH /api/create-event/{eventId} with buildCreateEventPayload(form).

- publishEvent(eventId):
  - ensure CSRF cookie.
  - POST /api/create-event/{eventId}/publish.

CreateEvent.vue API behavior:
- Save draft:
  - Calls persistDraft().
  - If draftId exists, it PATCHes the existing draft.
  - If draftId does not exist, it POSTs a new draft.
  - The response must return data.id for draftId.
  - lastSavedAt is updated.
  - draftSaved badge is shown briefly.

- Publish:
  - Validates steps 1, 2, and 3.
  - Calls persistDraft().
  - Calls publishEventRequest(draftId).
  - Stores response in publishedResponse.
  - Sets published to true, showing the success screen.

Expected publish response shape:
- data.id
- data.status
- data.public_url
- data.dashboard_url
- message

Success screen behavior:
- Shows "Your event is live!"
- Shows backend message or fallback: "Share it and start collecting attendees. Your dashboard can track everything live."
- Shows event ID/status if returned.
- Button routes to dashboard_url, public_url, or home.

30. BACKEND VALIDATION GUIDELINES
============================================================

The frontend validation is helpful but must not be trusted as the security boundary.

Backend should validate:
- User is authenticated.
- User owns or can manage the organiser account.
- Title exists and is within length limits.
- Event timezone is valid.
- Publish start time is at least 30 minutes in the future, calculated by backend time.
- End time is after start.
- Event type is one_time or recurring.
- Format is in-person, online, or hybrid.
- Venue is present for in-person/hybrid.
- Meeting link is present and a valid URL for online/hybrid.
- Category exists.
- Cover media exists for publishing.
- Recurrence frequency and pattern are valid.
- Weekly days are integers from 0 to 6.
- Monthly pattern is valid.
- Series end is present for recurring events.
- Occurrence count is positive when used.
- Recurrence duration does not overlap the next occurrence.
- Ticket names are present for paid ticket types.
- Prices are not negative.
- Money is stored as integer minor units.
- Ticket sales close after sales open.
- Ticket sales close no later than event start.
- Ticket inventory cannot oversell.
- Required attendee identity fields cannot be removed.
- Strings are sanitized/escaped on output to prevent stored XSS.
- Media uploads are validated by MIME type, size, and owner.
